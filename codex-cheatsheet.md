# Codex шпаргалка StartWave 🌊

## 1. Роль Codex в проекте

Codex — помощник для анализа, планирования и изменения файлов проекта.

Он может:

- изучить структуру проекта;
- найти ошибки и несоответствия;
- предложить план работы;
- изменить только указанные файлы;
- проверить результат;
- помочь подготовить изменения для Git.

Главный принцип:

> Сначала понять задачу → потом изменять.

Если задача ещё не определена точно, сначала лучше попросить Codex провести анализ без изменения файлов.

## 2. Хороший запрос для анализа

Пример:

```text
Проанализируй текущую структуру проекта.
Не изменяй файлы.
Покажи проблемы и предложения.
```

Такой запрос используется, когда нужно сначала понять текущую ситуацию.

Codex изучит проект и подготовит отчёт, но не будет вносить изменения. После анализа можно выбрать подходящее решение и дать отдельное подтверждение на работу с файлами.

## 3. Запрос на изменение файлов

Пример:

```text
Измени только эти файлы.
Не трогай HTML/CSS/JS.
После изменений покажи список изменений.
Не делай commit без подтверждения.
```

Такой запрос помогает сохранить контроль над работой.

Codex заранее понимает:

- какие файлы разрешено изменять;
- какие части проекта нельзя трогать;
- какой отчёт нужно показать после работы;
- можно ли создавать Git-коммит.

Чем точнее ограничения, тем меньше риск случайно изменить лишнее.

## 4. Ограничения задачи

В запросе всегда полезно указывать две группы правил.

### Что изменить

- конкретные файлы;
- конкретные разделы;
- ожидаемый результат.

Пример:

```text
Измени только PROJECT_SPEC.md и ROADMAP.md.
```

### Что не менять

- код;
- настройки;
- другие документы;
- Git-историю без подтверждения.

Пример:

```text
Не изменяй HTML, CSS, JavaScript и остальные документы.
Не делай git commit.
```

## 5. Проверка результата

После работы Codex нужно проверить изменения перед сохранением.

1. Посмотреть содержимое изменений:

   ```bash
   git diff
   ```

2. Проверить состояние проекта:

   ```bash
   git status
   ```

`git diff` показывает изменённые строки, а `git status` — список изменённых, новых и подготовленных к коммиту файлов.

Если результат правильный, изменения можно добавить в Git и сохранить отдельным коммитом.

## 6. Правильный рабочий цикл

```text
Идея
  ↓
Запрос Codex
  ↓
Анализ
  ↓
Подтверждение
  ↓
Изменение файлов
  ↓
git diff
  ↓
git commit
```

Этот цикл помогает не спешить и контролировать каждый этап.

Для большой задачи лучше сначала согласовать план, затем изменять файлы небольшими группами и проверять результат после каждого этапа.

## 7. Частые ошибки

### Ошибка: «Сделай всё сам»

Почему плохо:

- задача слишком широкая;
- непонятно, какие файлы можно менять;
- результат может отличаться от ожиданий;
- сложно проверить большое количество изменений сразу.

Лучше:

> Разделять работу на маленькие этапы.

Например: сначала анализ, затем документация, затем HTML, затем CSS и только потом проверка.

### Ошибка: «Измени проект»

Почему плохо:

- не указана цель;
- не указаны разрешённые файлы;
- отсутствуют ограничения;
- непонятно, как проверить готовность.

Лучше:

> Указывать конкретные файлы и ожидаемый результат.

Пример:

```text
Обнови только README.md.
Добавь краткое описание и ссылки на документацию.
Не изменяй код и другие файлы.
```

## 8. Примеры StartWave

### Реструктуризация документации

```text
Проанализируй PROJECT_SPEC.md, PROJECT_VISION.md и ROADMAP.md.
Сначала покажи, какие разделы нужно перенести.
Не изменяй файлы до моего подтверждения.
```

После согласования:

```text
Реструктурируй только PROJECT_SPEC.md, PROJECT_VISION.md и ROADMAP.md.
Не изменяй HTML, CSS и JavaScript.
После работы покажи список изменённых файлов.
Не делай git commit.
```

### Обновление Git-шпаргалки

```text
Обнови только cheatsheet-git.md.
Добавь простые объяснения git status, git diff, git add, git commit и git push.
Не изменяй другие файлы.
```

### Анализ PROJECT_SPEC.md

```text
Прочитай PROJECT_SPEC.md.
Не изменяй файл.
Покажи противоречия, недостающие требования и предложения по структуре.
```

Главное правило работы с Codex:

> Точная задача + понятные ограничения + проверка результата.
